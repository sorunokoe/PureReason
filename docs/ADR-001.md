# ADR-001: Scope, Small Classifiers, and Decoupling Strategy

**Decision Date**: 2026-04-28  
**Status**: Accepted  
**Author**: PureReason Project  
**Supersedes**: None (initial ADR)  

---

## Context

After fixing the vocabulary-elimination regressions (Priorities 1–3), the project faces three architectural questions:

1. **Small classifiers for logic**: Should we add fine-tuned small models (DistilBERT-level) to improve reasoning verification?
2. **Solver/Verifier decoupling**: Should arithmetic solver be decoupled from the hallucination detection core?
3. **Project scope**: What is the honest end-state scope—verifier, solver, detector, or ensemble?

This ADR clarifies the answers and establishes the architectural direction for the next phase.

---

## Decision 1: Small Classifiers Are Acceptable (TRIZ-aligned)

**Question**: Should we add fine-tuned small models (DistilBERT-level) for syllogism/reasoning verification?

### Answer: YES, with constraints

**Rationale**:

1. **Classifier approach is TRIZ-consistent**: The syllogism fix (Priority 2) already demonstrated that small TF-IDF+LogReg classifiers are acceptable:
   - F1 improved from 0.600 → 0.809 via classifier-first fallback chain
   - Classifiers are **data-driven, not hardcoded** — aligns with TRIZ principle "eliminate vocabulary in source"
   - Small models (< 100MB) preserve the project identity: "no LLM by default"

2. **Architecture constraint**: Classifiers must:
   - Be **deterministic and reproducible** (fixed seed, trained on committed artifact)
   - **Lazy-loaded** via `LazyLock` (no startup penalty)
   - Have **explicit provenance** (trained on which data, seed, version)
   - **Stay optional** (graceful fallback to heuristics if classifier missing)
   - Be **under 400 lines per classifier** (keep complexity low)

3. **Use cases for small classifiers**:
   - **Arithmetic chain verification**: 7-9 AST-based features, SVM or LogReg (low dimensional)
   - **Syllogism validity**: 15-20 NL features (spaCy tokens, premise complexity, entity overlap) — **already implemented**
   - **NL→logic template mapping**: Map question+premises → valid syllogism form (mood/figure) — experimental, not required

### Exception: No fine-tuned LLMs

The "small classifier" permission does NOT extend to fine-tuned LLMs (DistilBERT, RoBERTa, etc.):

- **Why not**: Fine-tuned LLMs require:
  - Substantial GPU infrastructure for retraining
  - Periodic retraining (drift risk)
  - Difficult reproducibility (PyTorch/HF version-lock required)
  - They blur the line between a verifier and a model-hosting product, which this repo is no longer pursuing

- **Alternative**: If semantic reasoning is needed, prefer:
  - Sentence transformers for embedding-based retrieval (frozen, reproducible)
  - Smaller symbolic models (SVMs, LogReg, naive Bayes on hand-crafted features)

## Decision 2: Solver and Verifier Remain Integrated (for now)

**Question**: Should arithmetic solver be decoupled into a separate subsystem?

### Answer: NO — Keep integrated (revisit at Scale 3)

**Rationale**:

1. **Current integration is justified**:
   - Arithmetic verification is a **sub-detector** within the hallucination-detection core
   - The solver (AST-based evaluation) and verifier (tolerance-based correctness) are tightly coupled
   - Decoupling would require:
     - IPC / service boundary (adds latency)
     - Separate versioning and deployment
     - Redundant state management (caching, serialization)

2. **When to reconsider** (Scale 3 transition):
   - If we add multi-language support (Python + JavaScript + Go solvers)
   - If arithmetic becomes a primary product (not a detector subcomponent)
   - If performance requires offloading solver to a separate service

3. **Current architecture is appropriate**:
   - Arithmetic is invoked in `crates/pure-reason-core/src/symbolic/` and `pureason/reasoning/chain.py`
   - Input/output are simple (expression string → result number)
   - Deterministic (same input → same output always)
   - Already testable in isolation (28 unit tests in `tests/test_arithmetic.py`)

## Decision 3: Project Scope is "Multi-Stage Verifier" (Honest Definition)

**Question**: What is the honest end-state scope?

### Answer: Multi-Stage Epistemic Verifier (not "detector" alone)

**New scope statement**:

> **PureReason is a deterministic multi-stage epistemic verifier for detecting
> hallucination-like patterns and verifying reasoning chains. It does NOT solve
> problems; it assesses confidence in solutions proposed by other agents.**

#### Clarifications

| Role | Scope | Module |
|---|---|---|
| **Solver** | ❌ Not in scope | Arithmetic evaluation is a means to verify, not a product |
| **Detector** | ✅ Core competency | Detects contradictions, misconceptions, semantic gaps |
| **Verifier** | ✅ Core competency | Validates reasoning chains (arithmetic, syllogisms, modus_ponens) |
| **Explainer** | ⚠️ Partial | Provides confidence bands + corrective suggestions (S49 rewrite) |
| **Optimizer** | ❌ Not in scope | Does not improve agent reasoning; only reports risk |

#### Architecture tiers (TRIZ Scale):

| Scale | Name | Stage |
|---|---|---|
| **Scale 1** | Heuristic verifier | ✓ Current (Mono → Bi transition) |
| **Scale 2** | Agent-facing verifier/runtime | Current direction |
| **Scale 3** | Ensemble (heuristic/symbolic/LLM with calibrated uncertainty) | Future (Poly) |
| **Scale 4** | Self-auditing (signed summaries, drift alerts) | Future (Field) |

## Immediate Implications

### 1. Documentation updates (CAPABILITIES.md, README.md)

**What to add**:
- Explicit "What PureReason is NOT" section
- Clarify solver vs verifier distinction
- Mention small classifiers as part of toolkit (not just Z3 logic)
- Remove any claims about arithmetic "solving" — frame as "verification"

### 2. Classifier governance

**New rule**: Any future classifier must include:
- Training script with reproducible seed
- Committed artifact (`.pkl` or `.joblib`)
- Provenance metadata (training data version, performance metrics)
- Fallback behavior if classifier unavailable
- < 400 lines of code

### 3. Decoupling roadmap

**For Scale 3 (future)**:
- If multi-language support is added, create abstract `Solver` interface
- Move solver logic behind trait boundary (allows pluggable backends)
- Defer actual decoupling until multi-language requirement is clear

## Trade-offs and Risks

### Risk: Classifier-based reasoning may seem "like cheating"

**Mitigation**: Classifiers are acknowledged as part of the toolkit. They're:
- Deterministic and auditable (unlike LLMs)
- Transparent in provenance (source, seed, performance published)
- Optional (graceful fallback to heuristics/logic)
- Subject to drift detection (holdout protocol in Priority 5)

### Risk: "No LLM by default" claim becomes muddied

**Mitigation**: Maintain clear terminology:
- "Small classifiers" = SVM, LogReg, TF-IDF, shallow neural nets (deterministic, < 100MB)
- "LLM" = the external frontier model already running in the calling agent
- "No LLM by default" remains true

### Risk: Scope creep into solver territory

**Mitigation**: Establish boundary:
- Verifier = "Given a solution, is it trustworthy?"
- Solver = "Find the solution" — out of scope
- Arithmetic solver is a **tactical exception** only; not a model for future systems
