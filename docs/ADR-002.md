# ADR-002: Agentic reasoning assurance roadmap

**Status**: PROPOSED
**Date**: 2026-04-30
**Authors**: Copilot + PureReason team  
**Context**: Follows ADR-001. Reframes PureReason from benchmark-centric
deterministic verifier work toward a broader agentic-era product strategy.

---

## Decision

PureReason should evolve into an **agentic reasoning assurance platform** that
**complements frontier models** instead of trying to beat them head-on as a
general reasoning model.

The first product shape should be a **hybrid**:

1. a reusable local assurance layer that existing agents can call through MCP, CLI, or loopback-local services
2. an optional shared control layer for traces, policies, review ops, and team governance
3. a flagship workflow that proves business value

The first flagship workflow should target **AI platform / engineering teams**:

- software plan/spec review
- software change review
- adjacent engineering assurance tasks where agent output must be checked before action

The commercial posture should remain **mostly open-source first**, with
enterprise value built on workflow assurance, governance, integrations, and
operations tooling.

---

## Why this direction

### External reality

Frontier models are increasingly powerful, but production agentic systems still
suffer from:

- non-determinism
- weak auditability
- cost and latency blowups
- tool misuse and long-horizon drift
- poor observability
- prompt injection and unsafe action execution
- hard-to-operate human review paths

### Internal reality

PureReason already has strong verification-oriented primitives:

- deterministic scoring and calibration
- contradiction and overconfidence checks
- arithmetic and logic verification
- rewrite/regulation functionality
- multiple integration surfaces (CLI, API, Python, MCP, dashboard)

What it does **not** yet have as a first-class product:

- durable agent state
- tool orchestration
- evidence/provenance systems
- review operations
- workflow-specific UX and ROI loops

That means the best path is to turn the current core into the **trust layer**
around agentic workflows.

---

## Strategic thesis

Do not build "a better frontier base model" first.

Build the system that makes frontier and local models usable in real workflows:

- integrate with the agent surfaces teams already use
- verify what the model planned or produced
- attach evidence and provenance
- block, escalate, or approve risky steps
- learn from failures and reviewer feedback

Do **not** rebuild the primary user-model chat loop when existing frontier agents
already provide it well.

This creates a product that is both more defensible and more aligned with the
current repo's strengths.

---

## Product principles

- **Verification first**: no high-impact action without an external check path
- **Agent-called first**: PureReason should be invoked by existing agents before it tries to own model interaction
- **Explicit state**: plans, tool calls, and approvals must be persisted
- **Agent agnostic**: support existing frontier agents through stable local contracts
- **Deterministic where it matters**: keep deterministic validators and policy engines
- **Open-source first**: drive adoption through usable core primitives
- **Zero-config local default**: local workflows should work without required API keys
- **Workflow anchored**: prove value in one painful workflow before broad platform expansion

---

## Target architecture

### 1. Agent-facing assurance surface

- MCP tools for Claude Code, Copilot, Codex, and similar agents
- local CLI review/verification contracts
- loopback-local API only when a local service is useful
- zero-config local operation by default

### 2. Shared control-plane services

- unified trace schema for tasks, reviews, findings, and decisions
- policy and review operations for teams
- local-to-team deployment path without owning model access

### 3. Agent runtime

- planner/executor/verifier loop
- task graph or state-machine execution
- retries, rollback, timeout, and termination rules
- state persistence and replay

### 4. Verification layer

- plan verifier
- tool-call verifier
- output verifier
- evidence grounding checks
- policy and compliance rules
- risk scoring and escalation logic

### 5. Memory and knowledge layer

- task memory
- evidence store
- organizational retrieval
- provenance and freshness tracking
- replay corpus for debugging and evaluation

### 6. Human operations layer

- review queues
- approval/escalation policies
- annotation and correction capture
- audit logs and postmortems

### 7. Learning loop

- capture failures and reviewer edits
- improve prompts, routing, and policies
- train compact specialist models only when workflow data proves value

## Roadmap stages

### Stage A - Truth reset

Goal:

- align product messaging with what the repo can really support

Deliverables:

- evidence-first documentation
- clean capability taxonomy
- explicit roadmap separating current product from aspirational strategy

### Stage B - Integrate with frontier agents well

Goal:

- make PureReason useful inside the agent surfaces teams already use

Deliverables:

- MCP and CLI review contracts
- zero-config local operation
- stable review/verifier response contracts
- task/trace wiring that external agents can depend on
- durable local review state (task, trace, and evidence persistence)

### Stage C - Build reliable execution

Goal:

- move from verifying single outputs to supporting real multi-step workflows

Deliverables:

- agent runtime
- tool contracts
- loop/stuck-state protection
- execution budgets
- replayable traces

### Stage D - Own one workflow

Goal:

- prove business value on engineering assurance work

Deliverables:

- software plan/spec review workflow
- software change review workflow
- reviewer UX and escalation rules
- ROI metrics tied to review reduction and incident avoidance

### Stage E - Build moat

Goal:

- use real workflow data to create durable advantage

Deliverables:

- failure corpus
- review outcome dataset
- evaluation packs from production traces
- compact specialist verifiers / rerankers / planners where justified

### Stage F - Platformize

Goal:

- turn the workflow engine into a broader product platform

Deliverables:

- SDK/API/MCP productization
- connector kit
- governance packs
- multi-tenant enterprise controls

---

## Success criteria

The core scorecard should move away from benchmark-F1-only thinking.

Primary product metrics:

- task success rate
- review load reduction
- time to safe approval
- incident / rollback reduction
- cost per completed workflow
- percentage of actions with verifiable trace and provenance

Secondary technical metrics:

- deterministic verifier correctness on targeted checks
- latency budget by workflow stage
- tool-call error rate
- model routing efficiency

---

## Non-goals

- training a frontier general-purpose base model as the first move
- competing directly on raw benchmark leadership as the main business story
- shipping a generic multi-agent platform before winning a concrete workflow
- replacing human approval in high-risk engineering actions too early

---

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Stay too generic | Tie the roadmap to software plan/spec/change review first |
| Overbuild orchestration too early | Start with a narrow planner/executor/verifier loop and strong contracts |
| Lose differentiation vs agent frameworks | Keep verification, auditability, and review ops as the center of the product |
| Open-source gives away too much | Keep the core open, but build moat in workflow data, governance, review ops, and enterprise integrations |
| Custom models become a distraction | Only train compact specialists after workflow data proves lift |

---

## Consequences

### Positive

- better fit with current repo strengths
- clearer wedge for AI platform and engineering teams
- more realistic path to business value in the agentic era
- stronger story than "we are the best reasoning model"

### Negative

- requires broadening from verifier-only thinking into workflow systems
- success depends on product discipline, not just benchmark work
- roadmap touches many layers: orchestration, verification, retrieval, and operations

## Appendix: Glossary

- **Scale 1**: Current deterministic verifier (heuristics + Z3)
- **Scale 2**: Hybrid layer with ensemble + optional classifiers
- **Scale 2b**: Scale 2 with distilled models enabled
- **Scale 2c**: Scale 2 with self-auditing loop enabled
- **Determinism**: Reproducible results (same seed → same output)
- **Explainability**: Reasoning chain provided (why ISSUE/OK decision)
- **Confidence**: Quantified uncertainty (0-1 score, higher = more certain)
- **Ensemble**: Multiple detectors voting on same question (diversity → robustness)
- **Distilled model**: Lightweight neural network (<20MB, <1B params)
